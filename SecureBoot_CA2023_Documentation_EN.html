Secure Boot CA 2023 AD Toolchain v4.0

A three-component toolchain for monitoring the Windows Secure Boot CA 2023 certificate update status across Active Directory-joined devices. No Intune, no Microsoft Graph API, no agent — data is collected via a GPO-deployed PowerShell script and visualised in a self-contained browser dashboard.

What is this toolchain?

The Windows Secure Boot CA 2023 update replaces ageing Secure Boot certificates on Windows devices. Microsoft is rolling out the update in phases from 2024 onwards; devices must meet BIOS version prerequisites before the certificate update can be applied.

This toolchain is designed for environments managed via Active Directory and Group Policy without Intune. It gives administrators the same level of status visibility as the native Intune Secure Boot Status report — covering CA 2023 rollout progress, registry state, event log diagnostics, and vendor BIOS readiness.

Architecture

The toolchain consists of three components that run independently:

1

Collector — Get-SecureBootCA2023StatusFull_v4.0.ps1

Deployed via GPO Scheduled Task to every domain-joined Windows device. Runs as SYSTEM and writes a per-device JSON file to a central network share. Collects CA 2023 registry state, event log events, OS version, hardware and BIOS information.

2

Aggregator — Merge-SecureBootCA2023Status_v4.0.ps1

Runs nightly via Task Scheduler on a management server with read access to the share. Reads all per-device JSON files, derives status groupings, and produces a single combined JSON and a flat CSV file.

3

Dashboard — SecureBoot_CA2023_Dashboard_v4.4.html

A self-contained HTML file opened locally in a browser. The combined JSON is imported via drag-and-drop or file picker. The dashboard performs the BIOS compliance table lookup live at import time — no data is baked into the JSON files.

Network share setup

Create a dedicated share on a file server or DFS namespace that is accessible by all domain-joined devices:

PrincipalShare permissionNTFS permission
Domain ComputersChangeCreate Files / Write Data only — no List or Read
Domain AdminsFull ControlFull Control
Service account (aggregator)ReadRead & Execute, List folder contents

Denying List folder contents to Domain Computers prevents devices from enumerating or reading each other's JSON files.

A Combined subfolder within the same share can be used as the aggregator output location, giving administrators a single UNC path to manage.

Collector script

The collector is a PowerShell 5.1 script that runs on each device and produces a JSON file containing the device's current CA 2023 status. It operates in three modes set by the $OutputMode variable at the top of the script.

GPO scheduled task

The repository includes a ready-to-use GPO XML file (SecureBoot-CA2023-StatusCollector_GPO.xml) that can be imported under Computer Configuration → Preferences → Control Panel Settings → Scheduled Tasks (type: Scheduled Task (At least Windows 7)).

1

Deploy the script

Copy Get-SecureBootCA2023StatusFull_v4.0.ps1 to a SYSVOL scripts folder or any UNC path readable by all domain computers (e.g. \\domain\NETLOGON\SecureBoot\).

2

Update the placeholder

Open the GPO XML and replace the *** CHANGE ME *** placeholder with the full path to the script, e.g.:
powershell.exe -NonInteractive -ExecutionPolicy Bypass -File "\\domain\NETLOGON\SecureBoot\Get-SecureBootCA2023StatusFull_v4.0.ps1"

3

Import the XML into the GPO

In the Group Policy Management Console, right-click the Scheduled Tasks node and select Paste after copying the XML content, or use the Import… option.

4

Link and configure the GPO

Link the GPO to the OUs containing the target computers. The task runs as SYSTEM, requires no credentials, and triggers daily at a configurable time plus on system startup. Set $CollectorShare in the script to your share UNC path before deployment.

Loading data into the dashboard

Open SecureBoot_CA2023_Dashboard.html locally in a browser. No web server is needed.

1

Obtain the combined JSON

Run the aggregator script to produce SecureBootCA2023_v4_Combined.json, or copy individual device JSON files directly to a local folder for smaller batches.

2

Import the file

Drag-and-drop the JSON file onto the dashboard, or click the upload area and use the file picker. Both a combined aggregator JSON (with a Devices array) and individual device JSON files are supported — multiple files can be loaded at once.

3

BIOS compliance is computed automatically

The dashboard looks up each device's model in the HP and Dell compatibility tables immediately on import. No additional steps are required. The result appears in the BIOS Ready column and the BIOS Compliance panel.

BIOS compliance check

The dashboard performs a live lookup of each device's model against vendor-published compatibility tables at import time. No BIOS compliance data is stored in the JSON files produced by the collector or aggregator.

ℹ️

To update the compatibility tables, edit the DELL_BIOS_TABLE and HP_BIOS_TABLE JavaScript constants in the dashboard HTML. No re-collection or re-aggregation is needed — all historical JSON files are instantly re-evaluated against the new tables on next import.

Model matching

Device model strings from WMI (Win32_ComputerSystem.Model) can differ from table keys due to marketing language, punctuation, and OEM configuration. The dashboard uses a three-stage matching pipeline:

1

Normalisation

Both the device model and every table key are lowercased, all non-alphanumeric characters replaced with spaces, and consecutive spaces collapsed. This makes punctuation and capitalisation irrelevant.

2

Stage 1 — Exact match

The normalised device model is compared directly against every normalised table key. If a match is found, it is used immediately.

3

Stage 2 — Token subset

Each table key's word tokens are checked against the device model's word tokens. If every token from the table key is present in the device model's token set, it is a match — regardless of additional words in the device string (e.g. “Inch”, “Notebook PC”, “2-in-1”). The longest matching key wins. A critical token guard rejects any candidate where a numeric or generation token (e.g. G5, 840) in the table key is absent from the device token set — this prevents cross-generation false matches such as G3 matching G5.

4

Stage 3 — Jaccard similarity

Jaccard word-token similarity is computed against all remaining candidates. The critical token guard is applied first, so generation-mismatched candidates are excluded before scoring. The best-scoring candidate above 60% is used. The match stage and score are recorded in the Match method field of the detail row.

Version comparison

BIOS versions are compared numerically, part by part (split on .). Each part is cast as an integer before comparison, handling leading zeros (01.10.00 → [1, 10, 0]) and differing part counts (02.67 vs 02.20.00). HP BIOS version strings may carry a model prefix (P01 Ver. 02.22); the extractor picks out the first digit.digit sequence.

Confidence Level values

Confidence Level is populated from the ConfidenceLevel registry value under HKLM:\SYSTEM\CurrentControlSet\Control\SecureBoot\Servicing (requires the November 2025 or later cumulative update) or from event log field extraction. Values are as documented in Microsoft KB 5016061 / KB 5068202.

ValueMeaning
High ConfidenceMicrosoft has high confidence the device will accept the update without issues. Automatic deployment recommended.
Under ObservationDevice is in a monitoring group. Deployment is cautious.
No Data ObservedInsufficient telemetry for this device or model.
Temporarily PausedDeployment paused for this device due to a known issue (see SkipReason / Event 1802).
Not SupportedDevice or configuration is not supported for this update.

Troubleshooting

ProblemCauseSolution
Status shows NO_EVENTS or UNKNOWN for all devices Devices have not received the November 2025 cumulative update Install the November 2025 or later CU; the registry keys and events only appear after this update
Reg Derived or UEFICA2023Error shows [object Object] PowerShell 5.1 ConvertTo-Json serialised a null-like registry value as {} Dashboard v3.0 automatically treats {} as null and displays —. No action needed; the Status token is still computed correctly from the event log.
BIOS Ready shows Not in List for a known HP or Dell device Model string from WMI does not match any table entry at 60% similarity or higher Check the Matched table key and Match method fields in the row detail. If no match was found, the model string can be added to the table in the dashboard HTML as an additional key.
Device shows a wrong BIOS table match Jaccard fuzzy match selected a similar but incorrect entry The critical token guard prevents cross-generation matches (G3 ≠ G5). If a wrong match is shown, add an exact key for the device model in the dashboard HTML.
JSON parse error on import Malformed or empty JSON file Check for parse errors reported by the aggregator summary. The aggregator skips and logs files that cannot be parsed.
All devices show as Stale Collector GPO not running or share path incorrect Verify the GPO is linked and applied (gpresult /r). Check that $CollectorShare in the script matches the actual share UNC path.
Model column is empty for all devices JSON files were collected with the v2.1 collector Re-collect with the v3.0 collector. Existing v2.1 files load correctly otherwise — BIOS compliance data is simply not available without the hardware fields.
Share write fails for devices Domain Computers lack Create Files permission on the share Verify NTFS permissions: Domain Computers need Create Files / Write Data. Read and List should be denied to prevent enumeration.
