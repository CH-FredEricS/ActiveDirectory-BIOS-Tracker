Secure Boot CA 2023 AD-Toolchain v4.0

Ein dreiteiliges Toolchain zur Überwachung des Windows-Secure-Boot-CA-2023-Zertifikat-Update-Status auf Active-Directory-Geräten. Kein Intune, keine Microsoft Graph API, kein Agent — Daten werden per GPO-Skript gesammelt und in einem selbstständigen Browser-Dashboard visualisiert.

Was ist dieses Toolchain?

Das Windows Secure Boot CA 2023-Update ersetzt veraltende Secure-Boot-Zertifikate auf Windows-Geräten. Microsoft rollt das Update seit 2024 schrittweise aus; Geräte müssen BIOS-Versionsvoraussetzungen erfüllen, bevor das Zertifikat-Update angewendet werden kann.

Dieses Toolchain ist für Umgebungen konzipiert, die über Active Directory und Gruppenrichtlinien ohne Intune verwaltet werden. Es gibt Administratoren dieselbe Sichtbarkeit wie der native Intune-Secure-Boot-Status-Bericht — einschließlich CA-2023-Rollout-Fortschritt, Registrierungsstatus, Ereignisprotokoll-Diagnose und BIOS-Bereitschaft des Herstellers.

Architektur

Das Toolchain besteht aus drei unabhängig laufenden Komponenten:

1

Collector — Get-SecureBootCA2023StatusFull_v4.0.ps1

Wird per GPO-Geplante-Aufgabe auf alle Domänen-Geräte ausgerollt. Läuft als SYSTEM und schreibt eine gerätespezifische JSON-Datei in eine zentrale Netzwerkfreigabe. Sammelt CA-2023-Registrierungsstatus, Ereignisprotokoll-Einträge, Betriebssystemversion sowie Hardware- und BIOS-Informationen.

2

Aggregator — Merge-SecureBootCA2023Status_v4.0.ps1

Läuft nächtlich per Aufgabenplanung auf einem Verwaltungsserver mit Lesezugriff auf die Freigabe. Liest alle gerätespezifischen JSON-Dateien, leitet Statusgruppierungen ab und erstellt eine kombinierte JSON- und eine flache CSV-Datei.

3

Dashboard — SecureBoot_CA2023_Dashboard_v4.4.html

Eine selbstständige HTML-Datei, die lokal im Browser geöffnet wird. Die kombinierte JSON-Datei wird per Drag-and-Drop oder Datei-Picker importiert. Das Dashboard führt die BIOS-Compliance-Tabellenabfrage live beim Import durch — keine Compliance-Daten werden in den JSON-Dateien gespeichert.

Eine dedizierte Freigabe auf einem Dateiserver oder DFS-Namespace einrichten, die für alle domänen-gejointen Geräte erreichbar ist:

Ein Combined-Unterordner innerhalb derselben Freigabe kann als Ausgabeort des Aggregators genutzt werden, sodass Administratoren einen einzigen UNC-Pfad verwalten.

Collector-Skript

Der Collector ist ein PowerShell-5.1-Skript, das auf jedem Gerät läuft und eine JSON-Datei mit dem aktuellen CA-2023-Status des Geräts erstellt. Der Betrieb erfolgt in drei Modi, die über die Variable $OutputMode am Anfang des Skripts gesteuert werden.

GPO-Geplante Aufgabe

Das Repository enthält eine fertige GPO-XML-Datei (SecureBoot-CA2023-StatusCollector_GPO.xml), die unter Computerkonfiguration → Einstellungen → Systemsteuerungseinstellungen → Geplante Aufgaben (Typ: Geplante Aufgabe (Mindestens Windows 7)) importiert werden kann.

1

Skript bereitstellen

Get-SecureBootCA2023StatusFull_v4.0.ps1 in einen SYSVOL-Skriptordner oder einen beliebigen UNC-Pfad kopieren, der für alle Domänen-Computer lesbar ist (z.B. \\domain\NETLOGON\SecureBoot\).

2

Platzhalter ersetzen

GPO-XML öffnen und den Platzhalter *** CHANGE ME *** durch den vollständigen Pfad zum Skript ersetzen, z.B.:
powershell.exe -NonInteractive -ExecutionPolicy Bypass -File "\\domain\NETLOGON\SecureBoot\Get-SecureBootCA2023StatusFull_v4.0.ps1"

3

XML in das GPO importieren

In der Gruppenrichtlinienverwaltungskonsole im Knoten «Geplante Aufgaben» rechtsklicken und Einfügen wählen, oder die Option Importieren… verwenden.

4

GPO verknüpfen und konfigurieren

GPO mit den OUs der Zielcomputer verknüpfen. Die Aufgabe läuft als SYSTEM, benötigt keine Anmeldeinformationen und wird täglich sowie beim Systemstart ausgelöst. Vor dem Ausrollen $CollectorShare im Skript auf den UNC-Pfad der Freigabe setzen.

BIOS-Compliance-Prüfung

Das Dashboard führt beim Import eine Live-Abfrage des Gerätemodells gegen herstellerveröffentlichte Kompatibilitätstabellen durch. In den vom Collector oder Aggregator erzeugten JSON-Dateien werden keine BIOS-Compliance-Daten gespeichert.

WertBedeutung

Modell-Matching

Modell-Strings aus WMI (Win32_ComputerSystem.Model) können von den Tabellenschlüsseln aufgrund von Marketingsprache, Satzzeichen und OEM-Konfiguration abweichen. Das Dashboard verwendet eine dreistufige Matching-Pipeline:

1

Normalisierung

Sowohl das Gerätemodell als auch jeder Tabellenschlüssel werden in Kleinbuchstaben umgewandelt, alle nicht-alphanumerischen Zeichen durch Leerzeichen ersetzt und aufeinanderfolgende Leerzeichen zusammengeführt. Dadurch werden Satzzeichen und Groß-/Kleinschreibung irrelevant.

2

Stufe 1 — Exakter Treffer

Das normalisierte Gerätemodell wird direkt mit jedem normalisierten Tabellenschlüssel verglichen. Bei einem Treffer wird dieser sofort verwendet.

3

Stufe 2 — Token-Teilmenge

Die Worttokens jedes Tabellenschlüssels werden gegen die Worttokens des Gerätemodells geprüft. Wenn jedes Token des Tabellenschlüssels im Token-Set des Gerätemodells vorhanden ist, gilt es als Treffer — unabhängig von zusätzlichen Wörtern im Geräte-String (z.B. «Inch», «Notebook PC», «2-in-1»). Der längste passende Schlüssel gewinnt. Ein kritischer Token-Guard verwirft Kandidaten, bei denen ein numerisches oder Generations-Token (z.B. G5, 840) im Tabellenschlüssel nicht im Geräte-Token-Set vorhanden ist — verhindert generationsbedingte Fehltreffer wie G3 auf G5.

4

Stufe 3 — Jaccard-Ähnlichkeit

Jaccard-Wort-Token-Ähnlichkeit wird gegen alle verbleibenden Kandidaten berechnet. Der kritische Token-Guard wird zuerst angewendet, sodass Kandidaten mit Generations-Mismatch vor der Bewertung ausgeschlossen werden. Der best-bewertete Kandidat über 60% wird verwendet. Match-Stufe und Score werden im Feld Match-Methode des Detailbereichs festgehalten.

Versionsvergleich

BIOS-Versionen werden numerisch, Teil für Teil (aufgeteilt an .) verglichen. Jeder Teil wird vor dem Vergleich als Integer umgewandelt, was führende Nullen (01.10.00 → [1, 10, 0]) und unterschiedliche Teilanzahlen (02.67 vs. 02.20.00) korrekt behandelt. HP-BIOS-Versionsstrings können einen Modell-Präfix enthalten (P01 Ver. 02.22); der Extraktor wählt die erste Ziffer.Ziffer-Sequenz.

Confidence-Level-Werte

Der Confidence Level wird aus dem Registrierungswert ConfidenceLevel unter HKLM:\SYSTEM\CurrentControlSet\Control\SecureBoot\Servicing (erfordert das kumulative Update vom November 2025 oder später) oder aus der Ereignisprotokoll-Feldextraktion befüllt. Werte wie in Microsoft KB 5016061 / KB 5068202 dokumentiert.
